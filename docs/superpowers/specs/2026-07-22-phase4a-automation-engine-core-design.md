# Phase 4A: Automation Engine Core — Design Spec

Sub-project 4A of `docs/PROJECT_PLAN.md`'s Phase 4 (Professional Automation & Growth Platform). This is the foundation every later Phase 4 sub-project (4B content generation, 4C release automation, 4D community monitoring, 4E notifications, 4F issue auto-response, 4G demo assets) builds on.

## Scope

4A ships the **draft-and-approve infrastructure** only: a `Draft` data model, a per-user REST API to list/approve/reject drafts, real-time cross-tab sync via the existing SSE `EventBroadcaster`, and a dashboard inbox reusing the exact Recommendations-inbox interaction pattern.

**Explicitly out of scope for 4A** (per `PROJECT_PLAN.md`'s own build-order table, each deferred to the sub-project that actually needs it):

- Any producer that generates real Draft content (README suggestions, release notes, social posts, issue replies) — that's 4B/4C/4D/4F. 4A's dashboard inbox will legitimately show its empty state until one of those lands.
- A generalized "named pipeline template" invocation abstraction. `PROJECT_PLAN.md` originally sketched this as part of "automation engine core," but with zero second pipeline yet to generalize against, building it now would be speculative (YAGNI — CLAUDE.md: "Don't design for hypothetical future requirements"). 4B is where the second stage sequence (the Agentic Content Pipeline) actually gets built; the right invocation shape will be obvious from that real second case, not guessed at now.
- Per-kind "on-approve" side effects (opening a PR, calling a social API, posting a GitHub reply). Approving a Draft in 4A is a pure status transition (`pending` → `approved`/`rejected`). Each future producer registers its own on-approve behavior when it lands — 4A only guarantees the row exists, is visible, and is approvable/rejectable in real time.
- Per-user APScheduler jobs beyond the existing daily analytics run. No feature needs one yet.

This keeps 4A a genuinely reviewable, testable, shippable increment on its own, matching the user-approved "one sub-project at a time" build order.

## Data model

New table, one migration, autogenerated via `alembic revision --autogenerate` from `app/models.py` (this project's established workflow — see `alembic/versions/0001_initial.py`).

```python
class Draft(Base):
    __tablename__ = "drafts"

    id: Mapped[int] = mapped_column(Integer, primary_key=True)
    user_id: Mapped[int] = mapped_column(ForeignKey("users.id", ondelete="CASCADE"))
    repo_id: Mapped[int | None] = mapped_column(ForeignKey("repos.id", ondelete="CASCADE"), nullable=True)
    kind: Mapped[str] = mapped_column(String(100))
    target: Mapped[str] = mapped_column(String(255))
    content: Mapped[dict] = mapped_column(JSON)
    status: Mapped[str] = mapped_column(String(50), default="pending")
    created_at: Mapped[datetime] = mapped_column(DateTime(timezone=True), default=_now)
    reviewed_at: Mapped[datetime | None] = mapped_column(DateTime(timezone=True), nullable=True)
```

- `repo_id` nullable: some future drafts (e.g. an account-level community-monitoring digest) aren't tied to one repo.
- `content` is `JSON`, not `Text` — `Recommendation.body` is free text because it only ever holds prose; `Draft.content` must hold structurally different shapes per `kind` (a diff for a README suggestion, a `{platform, text}` pair for a social post), so a JSON blob is the right fit, same reasoning as `BenchmarkRepo.topics`.
- `kind`/`target`/`status` are plain `String` columns, no DB-level enum — matches the existing convention (`Recommendation.category`, `PipelineRun.status` are unconstrained strings too); validity is enforced at the Pydantic API layer instead, consistent with the rest of this codebase.
- No `Draft` values exist for `kind`/`target` yet since no producer exists in 4A. The API layer validates `status` transitions only (`pending`→`approved`/`rejected`); it does **not** validate `kind` against an enum, since the valid set is still empty and will grow with each future sub-project — over-constraining it now would mean editing this table's constraint on every future sub-project, which is exactly the kind of premature rigidity YAGNI warns against.

## Backend API

New router `app/api/drafts.py`, mirroring `app/api/recommendations.py` byte-for-byte in structure (same `require_api_key`+`require_user` dependency chain, same 404-not-403 per-user-scoping pattern, same `from_attributes` Pydantic config).

```text
GET   /drafts                    -> list[DraftOut], current user's drafts, newest first
PATCH /drafts/{draft_id}         -> DraftOut, body: {"status": "approved" | "rejected"}
```

- `PATCH` validates `status` is exactly one of `"approved"`/`"rejected"` (Pydantic `Literal["approved", "rejected"]`) — attempting to set `"pending"` or any other value is a 422, not silently accepted.
- `PATCH` on a `Draft` that isn't `"pending"` (already reviewed) returns `409 Conflict` — approving/rejecting is a one-way transition, not a toggle (unlike `Recommendation.dismissed`, which the existing UI does let you flip back and forth). This is a deliberate difference: a Draft's `reviewed_at` is meant to be a real audit trail of "when was this decided," which a re-flippable status would undermine.
- On successful `PATCH`, sets `reviewed_at = now()`, commits, publishes `draft_updated` via the existing `broadcaster.publish(..., user_id=current_user.id)`.
- Endpoint path `drafts` contains none of the forbidden substrings (`analytics`/`analysis`/`tracking`/`performance`/`metrics`) — compliant with the ad-blocker-safe naming rule.
- Router registered in `app/main.py` alongside the other 6 routers, same `dependencies=[Depends(require_api_key)]` pattern.

No `POST /drafts` endpoint in 4A — nothing creates drafts yet (no producer exists), so an unused creation endpoint would be dead API surface. `Draft` rows for the end-to-end tests below are seeded directly via the ORM in tests, exactly like `Recommendation` rows already are in `test_recommendations_api.py`.

## Real-time sync

Reuses the existing `EventBroadcaster`/SSE infrastructure exactly as-is — no changes to `app/events.py` or `app/api/events.py`.

- New event type: `draft_updated`, published on every successful `PATCH /drafts/{id}`.
- Frontend `EVENT_QUERY_MAP` (`hooks/use-live-events.ts`) gains one entry: `draft_updated: [queryKeys.drafts.all]`.
- This guarantees the standing cross-tab/no-refresh requirement: approving a draft in one tab instantly updates the Drafts inbox in every other open tab for that same user, with zero page reload — identical mechanism to how `recommendation_updated` already works today.

## Frontend

Mirrors the Recommendations page/hook/component trio exactly:

- `lib/query-keys.ts`: add `drafts: { all: ["drafts"] as const }`.
- `lib/api-types.ts`: add `export type Draft = components["schemas"]["DraftOut"];` (generated from the live OpenAPI schema via `npm run generate:types` — never hand-written, per CLAUDE.md).
- `lib/api.ts`: add `listDrafts: () => backendFetch<Draft[]>("/drafts")` and `reviewDraft: (id: number, status: "approved" | "rejected") => backendFetch<Draft>(\`/drafts/${id}\`, { method: "PATCH", body: JSON.stringify({ status }) })`.
- `hooks/use-drafts.ts`: `useDrafts()` (query) + `useReviewDraft()` (mutation, `setQueryData` cache update on success, matching `use-recommendations.ts`'s pattern — not a full refetch).
- `app/drafts/page.tsx`: Server Component, `export const dynamic = "force-dynamic"`, `Promise.all([api.listDrafts(), api.listRepos()])` prefetch (repos needed to resolve `repo_id` to `owner/name` for display, same as Recommendations does), hydrates a `DraftsClient`.
- `components/drafts/drafts-client.tsx`: Client Component. Same shape as `recommendations-client.tsx` — `SectionHeading` (new icon: `Inbox` from `lucide-react`, color `text-emerald-500` — not yet used by any other nav/section icon, verified against `nav-sidebar.tsx`'s existing `NAV_ITEMS` colors: sky/amber/violet/slate/rose), `EmptyState` for the zero-drafts case ("No drafts yet — they'll show up here once an automation feature generates one."), one `Card` per pending draft showing `kind`, resolved repo name (or "Account-level" if `repo_id` is null), and two actions instead of one: an approve button (`CheckCircle2`, `text-emerald-500`) and a reject button (`X`, `text-red-500`), each calling `useReviewDraft().mutate(...)` with an `onError` toast, each `disabled` while `isPending`. Only `status === "pending"` drafts render (approved/rejected drop out of view, same as dismissed recommendations do).
- `components/nav-sidebar.tsx`: add one `NAV_ITEMS` entry — `{ href: "/drafts", label: "Drafts", icon: Inbox, color: "text-emerald-500" }`, positioned after "Recommendations" (both are review-inbox-shaped pages, so they stay adjacent).
- `hooks/use-live-events.ts`: add the `draft_updated` mapping described above.

No new UI primitives needed — `Card`, `CardContent`, `Button`, `EmptyState`, `SectionHeading` all already exist in `components/ui/` and are reused as-is.

## Testing

Backend (`backend/tests/test_drafts_api.py`, new file, mirroring `test_recommendations_api.py`'s structure):

- `GET /drafts` returns only the calling user's drafts (seed two users, two drafts, assert isolation) — same cross-tenant-isolation pattern already proven for every other resource.
- `PATCH /drafts/{id}` with `{"status": "approved"}` on a pending draft: 200, `status` becomes `"approved"`, `reviewed_at` is set. `test_recommendations_api.py` does not assert on SSE publish at all (checked — no existing precedent), so add one new assertion here via `unittest.mock.patch("app.api.drafts.broadcaster.publish")`, confirming it's called once with `("draft_updated", {"id": ..., "status": "approved"}, user_id=...)`.
- `PATCH` with an invalid `status` value (e.g. `"pending"` or `"deleted"`): 422.
- `PATCH` on an already-`approved` draft: 409, `status` unchanged.
- `PATCH` on another user's draft: 404 (not 403 — matches every other resource's existence-leak-avoidance pattern already established, e.g. `require_user`'s generic-message convention).
- `PATCH`/`GET` without a valid internal user token: 401 (via the `client_without_user_token` fixture already in `conftest.py`).

Frontend (`frontend/tests/`, new file(s) mirroring the existing 4 test files' structure — check `tests/internal-auth.test.ts` or similar for the project's Vitest conventions before writing):

- `useReviewDraft`'s mutation correctly updates the query cache on success (mirrors any existing mutation-hook test, or add one if none currently exists — check first).
- `EVENT_QUERY_MAP` includes `draft_updated` mapped to `queryKeys.drafts.all` (extend the existing `use-live-events` test if one exists).

Both suites must stay at 100% pass, zero warnings, matching the rest of this codebase (already re-verified clean in the pre-Phase-4 audit immediately preceding this spec).

## Migration & type-generation sequencing

Order matters for this task, unlike most: the OpenAPI-generated frontend types depend on the backend actually running with the new endpoint.

1. Add `Draft` model, run `alembic revision --autogenerate`, review the generated migration for correctness (per this project's established manual-review-after-autogenerate habit), `alembic upgrade head` against local dev Postgres.
2. Build and test the backend API fully (`GET`/`PATCH /drafts`) in isolation first.
3. Start the local backend (`uvicorn app.main:app`), run `npm run generate:types` from `frontend/` to regenerate `types/api.d.ts` with the new `DraftOut` schema.
4. Build the frontend layer (types → api.ts → query-keys → hook → page → client component → nav link → SSE mapping).
